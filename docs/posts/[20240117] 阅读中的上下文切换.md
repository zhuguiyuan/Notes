# 阅读中的上下文切换

某些事情的时间跨度非常的长。
比如阅读这件事情，面对一本两百页的专业书，一天读完也是很困难的。
更不要说那些六百页的大部头或者一千页的巨著了。
可能在一章之后，我们就需要停下来消化吸收。
然后做一些别的事情，比如喝杯水，赶一赶组会工作进展汇报的 DDL。
于是，一本书被放下了。或许就再也不会拿起来，或许再次拿起来又会重复这一过程。

<!-- more -->

一些阅读的技术可以帮助缓解这个状况。
例如，《如何阅读一本书》中认为我们可以使用检视阅读，
包括系统化的粗读和囫囵吞枣的粗浅通读，来在一个比较短的时间获取对一本书讨论主题的理解。
系统化的粗读是选取书本中的一些部分进行阅读；
囫囵吞枣的粗浅通读是一种快速的通读：
阅读的进度是一直向前的，跳过难以理解的段落，不要返回看过的段落
（即使没有理解也不要返回，如果想要反复，那么不如试着通过放慢阅读的速度来消除反复）。
如果时间允许，花一整个下午或许可以对一本两百页的书完成囫囵吞枣的粗浅通读，
但是更多情况是只有四十五分钟，使用系统化粗读熟悉书本的梗概。

检视阅读并没有解决长时间跨度的问题，它只是让短时间内能够理解和吸收的东西增多了。
就像是把一个 FPGA 虚拟化，各个虚拟化的 vFPGA 租用给不同的应用同时使用，降低了资源碎片率；
但这并没有解决大规模的负载在一个 FPGA 上放不下的问题。

事情的转机发生在过去两年的刷课过程中。
为了补全 CS 的基础知识，我跟着 csdiy.wiki 在学习 UCB 的 CS61ABC 三门课程。
由于中途有许多的事情（考研、调剂、上课、科研）打断
，我用 markdown 在一个文件里面用代办列表记录下了我的刷课进度。
虽然完成这三门课的时间跨度长达两年，但他们却奇妙地持续下来了。
这种持续是来自每次打断的信息记录和每次继续时的现场恢复。
标题中的词是上下文切换，是的，这就像是操作系统中的进程上下文切换。
我会用类似记录新闻的手法记录为视频记录下笔记，标记上日期；
用一个文件维护需要做的事情（第几周的第几个视频，第几个 lab 和 project）。
这个进度列表帮我知道应该从哪里恢复，这些笔记帮我拾起当时的记忆。

这种上下文切换也可以应用在阅读中。
我们可以拿一张草稿纸写下上一次阅读的一些场景，感受，和体验。
然后把这张“读到了这里”的书签夹在截止的页上。
或者也可以在这张纸上添加一个进度列表。
对于纸书是这样，触摸着实质性的进度纸张可以帮助我们回到上一次停下的地方。
对于电子书，我的阅读形式是使用 Goodnotes 阅读 PDF 文件。
Goodnotes 有书签功能，可以作为阅读进度的里程碑。
我也可以在文档中写写画画，记录下思想的上下文。

长时间的阅读相比于花一天时间，撂下其他的活，沉在书里，是一种更可持续的形式
（或许也有助于健康的生活方式，对抗赶 DDL 的习惯）。
在后者中人会很容易疲倦，或许只有前是30 页是有效的阅读。
而前者，在回顾上下文的时候，我们也自动地获取了一点复习的感觉。
对书籍所讨论的主题的理解在波浪式地演进。
所以，当遇到需要长时间跨度阅读的书的时候，不妨试试这项上下文切换的技术。
